==================================================
2) PROJEKT-KURZÜBERBLICK
==================================================
syn4ps3h0me ist ein lokales Smart-Home- und Homelab-Setup (Raspberry Pi Fokus), das eine komplette Basisplattform für
Monitoring, Messaging, DNS/Adblocking, Reverse Proxy, lokale KI und Voice-Automation bereitstellt.

Zentrale Eigenschaften:
- Einfache Installation über install.sh
- Docker-Compose als Betriebsgrundlage für Kernservices
- Monitoring-Stack (Telegraf + InfluxDB + Grafana)
- MQTT-Broker (Mosquitto)
- DNS/Adblocking (Pi-hole)
- HTTPS-Zugriff/Reverse Proxy (Caddy)
- Open WebUI für lokale KI-Interaktion inkl. RAG
- Voice-Pipeline mit Wake-Word, lokaler STT/LLM/TTS und Shelly-Integration

==================================================
3) SYSTEMARCHITEKTUR (LOGISCHE SICHT)
==================================================
3.1 Infrastruktur-/Core-Services
- Mosquitto: MQTT Nachrichtenbus
- InfluxDB: Zeitreihenspeicher
- Telegraf: Datensammlung und Ingestion in InfluxDB
- Grafana: Dashboards/Visualisierung
- Pi-hole: DNS + Werbeblocker + optional DHCP
- Caddy: Reverse Proxy + TLS Terminierung

3.2 KI-/Interaktions-Layer
- Open WebUI: Chat-Frontend, Modellzugriff, Dokumente/Knowledge/RAG
- Voice-Pipeline:
  - Wake-Word Erkennung
  - Whisper Speech-to-Text
  - Command-Routing (Smart-Home vs. LLM)
  - Shelly REST-Steuerung
  - TTS Ausgabe

3.3 Persistenz & Konfiguration
- Docker volumes für persistente Servicedaten
- Servicekonfigurationen in jeweiligen Unterordnern (z. B. telegraf/, grafana/, pihole/, caddy/, voice-pipeline/)
- Secrets/Passwörter über .env

==================================================
4) OPEN WEBUI + RAG BETRIEBSMODELL (WICHTIG)
==================================================
4.1 Wissensordner in diesem Projekt
- Host-Pfad: open-webui/knowledge/
- In den Container read-only gemountet nach: /app/backend/data/knowledge-import

4.2 Erwartetes Verhalten gemäß Open WebUI RAG-Konzept
- Dateien im knowledge-import-Ordner sind als Importquelle verfügbar.
- Für Knowledge-Bases erfolgt die tatsächliche Indexierung/Einbindung innerhalb Open WebUI (Workspace > Knowledge).
- Danach kann Wissen im Chat per Knowledge-Auswahl oder Referenz genutzt werden.

4.3 "Out of the box" Betriebsziel
- Nach Start des Compose-Stacks ist der Ordner sofort im Container vorhanden und kann ohne zusätzliche Host-Mounts verwendet werden.
- Für RAG-Qualität sind Chunking/Embedding-Einstellungen in Open WebUI Admin Settings (Documents) relevant.

4.4 RAG-Qualitätsempfehlungen
- Strukturierte .txt/.md Dokumente mit klaren Überschriften verwenden
- Thematisch getrennte Dateien statt einer riesigen Monolith-Datei
- Bei großen Dokumenten auf Chunk Size, Chunk Overlap und Chunk Min Size Target achten
- Bei Ollama-Modellen ausreichend Context Length konfigurieren

==================================================
5) INSTALLATION, BOOTSTRAP & BETRIEB
==================================================
5.1 Bootstrap-Skript
- install.sh übernimmt große Teile der Erstinstallation:
  - Systempakete
  - Docker
  - Hailo-bezogene Vorbereitungen
  - ggf. service setup (hailo-ollama)
  - Start des Compose-Stacks

5.2 Compose-Laufzeit
- docker-compose.yml definiert alle zentralen Services, Netzwerke und Volumes.
- Die Services sind grundsätzlich auf Dauerbetrieb (restart unless-stopped/always) ausgelegt.

5.3 Secrets / Umgebungsvariablen
- Kritische Zugangsdaten werden über .env bereitgestellt.
- Für Produktivbetrieb müssen alle relevanten Passwörter/Tokens gesetzt sein.

==================================================
6) DATENFLÜSSE (PRAKTISCHE SICHT)
==================================================
6.1 Monitoring-Datenfluss
Sensor/Host/MQTT -> Telegraf -> InfluxDB -> Grafana Dashboards

6.2 DNS/Web-Zugriff
Clients -> Pi-hole (DNS, Filtering) -> interne Namensauflösung
Browser/Clients -> Caddy -> interne Services (HTTPS/Proxy)

6.3 Voice-/KI-Datenfluss
Mikrofon -> Wake-Word Trigger -> Audioaufnahme -> Whisper STT -> Router:
- Smart-Home Intent erkannt -> Shelly REST command
- Sonst -> LLM Request -> Antwort -> TTS -> Audioausgabe

6.4 Open WebUI RAG-Datenfluss
Wissensdatei (open-webui/knowledge) -> Import in Workspace/Knowledge -> Embeddings/Index -> Chat Retrieval

9) BETRIEBS- UND TROUBLESHOOTING-CHECKLISTE
==================================================
9.1 Grundchecks
- docker compose ps
- docker compose logs -f <service>
- systemctl status hailo-ollama (falls verwendet)

9.2 Open WebUI / RAG
- Prüfen, ob knowledge-Ordner im Container sichtbar ist
- Dokument in Workspace > Knowledge importieren
- Bei schlechter Retrieval-Qualität:
  - Chunking-Parameter prüfen
  - Embedding-Modell prüfen
  - Reindex ausführen (falls Embedding geändert)

9.3 Voice-Pipeline
- Audio-Devices/Permissions prüfen (/dev/snd, audio-Gruppe)
- Wake-Word-Schwellenwert anpassen
- Whisper-Modell/Language Settings prüfen
- Shelly Device Map Pfade/URLs validieren

9.4 Monitoring
- Telegraf Inputs erreichbar?
- InfluxDB Token/Bucket/Org korrekt?
- Grafana Datasource healthy?
